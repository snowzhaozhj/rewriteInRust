# 模块拆解粒度重设计

> 状态：已定稿（grilling 访谈 + 两轮 Codex 对抗审查 + 用户终审通过）。
> 一句话：把"每个源文件=一个翻译单元、各跑完整重型流水线"改为"按上下文预算合并机械小文件 + 用规模/危险信号分流轻重，翻译前先在真实项目上量化验收拆解是否合理"。

## 1. 背景与问题

现状：翻译单元 = 单个源文件（循环依赖已折叠为 SCC composite，其余一文件一模块）。每个单元不论大小都套同一套重型流水线（意图摘要→Phase A 多候选→对抗审查/选优→Phase B→测试→签批，standard/full 档 4-5 次 subagent 冷启动往返）。

后果：一个 3 函数、<10 行的小文件被拆成独立模块，跑完整流水线约半小时。**主导成本 = subagent 往返次数 × 冷启动延迟**（已确认，非编译、非病态循环为主）。

`往返总数 ≈ 单元数 × 每单元流程深度`。现状两个维度都失控：单元数 = 文件数（小文件碎片化），深度 = 不分轻重（二值 danger token：现状 `if_statement`/`Math.`/`try` 任一出现就判最高档 Full，见 `typescript.rs:243`）。

## 2. 根本诊断

两个错误叠加：

1. **粒度错配**：固定开销（subagent 冷启 + 整 workspace 编译 + 多次 state CLI 往返）远大于小文件的实际翻译量。
2. **分类太粗**：复杂度判定是"内容种类二值"，完全无视规模——10 行带一个 `if` 的文件，和 2000 行并发模块，吃同一套最重流水线。

目标精确化：**只除"小而机械被过度处理"这一类不该付的开销，不误伤"小而难/危险"的文件**（后者的重型成本是在抓真陷阱，属于该付）。

## 3. 范围边界（先划清，避免越界）

本设计**只解决源码侧"怎么分组翻、怎么分轻重"**，目标是砍掉小文件的过度开销。以下**明确不在本设计范围**：

- **Rust 目标侧的合并冲突**：跨并行兄弟的 trait coherence(E0119) / orphan rule / feature 门控 / 宏展开 / Cargo.toml·lib.rs 合并冲突。这些边**不在源码 import 图里**（源码图只有 File 节点 + Imports 边），拆解再凸、内聚再高也预测不了。
- 这类冲突**由现有架构在翻译执行期兜底**：并行编排的"整组编译真门 + compile_fixing 子流程 + reconcile"（见 `run.md` 并行编排两层 done），不是分组期能预防的。
- **翻译质量本身**：本设计不验翻译对不对，只验"拆解合不合理"。

承认这条边界很重要：它解释了为什么我们**不**做"keep-together 架构单元"（见 §9），也限定了验收门的声称范围（见 §8）。

## 4. 核心原则

### 4.1 CLI / LLM 分工
- 确定性事实 + 机械计算 → CLI（图、复杂度信号、合批、footprint、不变量强制）。
- 语义判断 / 实际翻译 → LLM。
- **可复现性靠"冻结拆解计划"（确定性算出 + 持久化），不靠排除 agent。** 断点续传照冻结的计划走（需新增 state 字段，见 §7）。

### 4.2 两个独立决策（互不化简）
| | ① 单元多大 | ② 单元跑多重流程 |
|---|---|---|
| 解决 | 小文件别各自付固定开销 | 简单的别跑重型流水线 |
| 驱动 | **footprint（上下文规模）硬约束** | **是否"可证明机械"** |
| 性质 | 硬物理约束（装不下=翻不了） | 软质量诉求 |

两轴正交，正好从两个方向砍主导成本（单元数 × 深度）。

## 5. 决策②：流程深浅 + 危险信号（厘清，避免自相矛盾）

两件**不同**的事，别混：

**(a) 流程深度 = 二分**：
- "可证明机械"（纯类型/常量/barrel + 无危险信号 + 小）→ **轻流程**。
- 其余 → **重流程**（保守，宁可多花别翻错）。
- 不做多轴难度打分：难度无法可靠静态预测（调研结论：传统复杂度 ≠ LLM 翻译难度），只能可靠区分"机械 vs 其余"，到此为止。

**(b) 危险信号 = 规则注入 + 定向测试（独立于深度）**：
- 强信号类别：数值精度（Math/浮点/Number）、并发（async/Promise.all/线程）、动态（eval/反射/getattr/monkeypatch）、IO 副作用、FFI、跨文件共享可变全局。
- 命中危险信号 → **① 该文件即"非机械"（因此走重流程）+ ② 注入对应 porting 规则（复用项目已有 RULE-3 整数溢出 / RULE-22 异步运行时等机制）+ ③ 加该陷阱的定向测试。**
- 厘清"危险信号不调重流程"的原意：**重型流水线本身不是抓 NaN 的手段**（它是钝器，要审查者知道去探才有用）；真正抓陷阱靠"规则注入告诉模型这个坑"。危险文件该走重流程——但那是"非机械"决定的，规则注入是额外的、针对性的一刀。

## 6. 决策①：单元大小（footprint）

`footprint = 自身源码字符 + 实际用到的依赖符号签名规模`。

- 依赖注入策略（已定）：**1-hop 直接依赖 + 只喂 target 实际用到的符号**。漏掉的传递类型由 F1 编译反馈环自我修正，不靠预注入全部兜底。
- **现状不支持，需新建**（Codex 核实）：named import 符号在解析期有（`ImportInfo.symbols`），但 `build_graph` 只折成文件级 `Imports` 边、未保留"用了哪些符号"；`graph interfaces --deps-of` 返回依赖全量 exports 且无签名规模字段。故 PR-1 要：① 把"target→各依赖的被用符号集"做成可查（持久化到图或新增查询）；② `graph interfaces --deps-of` 按被用符号裁剪 + 输出签名字符规模；③ footprint 估算器。
- 按 footprint 三段定粒度：远低于预算 → 合并；落预算带内 → 单独；超预算 → **标记转人工**（自动子文件拆分推迟）。

## 7. 分组算法 + 两种 composite

简单凸性拓扑 first-fit：

1. condensation 折叠循环依赖（现有 SCC，petgraph）。
2. 标记可合批节点 = 严格机械 + 小。
3. DAG 上按确定性拓扑序（字典序破平）连续装箱，满 footprint 预算封口；保证**凸性**（合并不得制造跨外部节点的回边 → 不成环、batch 内部必无环）。
4. size>1 的 batch → composite 单元；size==1 → 单文件模块。

**两种 composite 必须区分（Codex 抓的关键遗漏）**：现状 `member_files` 非空一律被当 SCC 组、走契约门+逐文件填空的重型路径（`run.md:75`）。本设计的机械合批组**不该**走那条路。故：

- composite 带类型标记（复用已有 `is_cycle`：循环组 vs 合批组）。
- **循环组** → 现有契约路径（不变）。
- **机械合批组** → **新增轻量路径**：一个 subagent 按逆拓扑序一次翻完整批 + 一次整组编译 + 一次签批。因为成员机械、无环、footprint 受控，无需契约/stub/逐文件填空。run.md 要加这条分支。

**轻量路径的状态机接入（补规范，回应 Codex B2）**：机械合批组**作为单模块整体 dispatch**（不复用 SCC 的 `contract_ready`/`phase_a_in_progress`/`contract_revision` substatus，也不写 `{group}-progress.json` 逐成员 checkpoint）。它走**单文件模块同款的简单状态流**（`translating → done`），锁/dispatch 机制复用单模块。**checkpoint = 单元自身 status；恢复 = 整批重新 dispatch**——因为是"一次翻完"的原子操作，无需成员级断点（整批一次的成本本就低，重跑即可）；已落盘且签名校验通过的可按 content-hash 跳过。run.md 断点路由表需为"合批组"加这一条恢复入口。

**PR-1/PR-2 顺序安全（回应 Codex B1）**：PR-1 产出合批 composite，但 PR-2 才有轻量路径。两者之间若误触 `/migrate run`，合批组会按现状误走 SCC 契约重路径。故 **PR-1 同步加守门**：`/migrate run` 遇到"合批组类型"的 composite 而轻量路径未就绪时**显式报错**（"机械合批轻量路径尚未启用，请勿 dispatch"），不静默走 SCC。PR-1 阶段合批 composite 只被 dry-run 报告消费、不进 active dispatch（验收活动也只读 dry-run）。

可复现：拆解计划（单元划分 + 分类）确定性算出后**冻结落盘**；断点续传以冻结计划为准、不重算。**需新增 state 字段**（计划 content hash + 冻结标记；现状 `populate-modules` 每次重算、`ModuleState` 无此字段）。输入用稳定 key（repo 相对路径 + content hash）。

> 内聚加权打包（沿符号耦合重边优先合并、在弱缝切）= 待定优化，由验收门"跨 bin 边占比"数据触发，不预建。

## 8. 验收门（锁定，翻译前的硬前置）

翻译之前，只跑拆解、出可量化报告，判定**源码侧拆解**是否合理。**不派任何翻译 subagent，不声称验证 Rust 侧 / 翻译质量**（见 §3 边界）。

| 维度 | 指标 | 算法 |
|------|------|------|
| 目标达成 | 模块数 before(1文件=1模块) vs after；缩减率；被合批文件占比；**残留机械单文件模块数（应≈0）** | 纯计数，CLI |
| 正确性不变量（硬门 100%） | 模块 DAG 无环；每 batch 凸；每文件恰好归属一次；**跑两次字节级一致**；无 batch 超预算 | 纯图/计划，CLI |
| 内聚质量 | bin 内边 vs 跨 bin 边占比；与随机基线对比 | 见下 |
| 分类合理性 | 机械/重型/footprint 分布；锚点反例 | CLI 出分布；**人工抽检**判语义 |

**内聚 MQ 算法（补定义，回应 Codex；硬门必做、不可跳过）**：对每个 batch，`内部边 = 成员间符号耦合边数`，`外部边 = 成员到 batch 外的符号耦合边数`；batch 内聚 = `内部/(内部+外部)`；全局取加权平均。随机基线 = 固定 seed 下、保持各 batch 大小不变随机重排成员，算同一指标，取 N 次均值；要求实际高于随机基线（首版用简单阈值"≥1.5×"，不上正式显著性检验，避免过度工程）。**这是硬门四维度之一，不允许"待定"跳过**——否则无法排除"合批只是按拓扑序机械凑数、内聚极低"。简化算法（内部/外部边比）实现成本低，首版即落。

**分类合理性是 CLI + 人工**：CLI 出分布 + 自动校验锚点反例（"10 行带 if" 不因 if 进重型 / "3 行 clamp 带 Math" 命中危险信号）；语义一致率靠人工抽检 ≥80%（这一项本就需人类标注，不是纯 CLI，明示）。

成功判据：不变量+确定性 **100%**；残留机械单文件 ≈0；内聚 ≥1.5× 随机基线（硬门，不可跳过）；锚点反例自动校验 100%；人工抽检 ≥80%。模块缩减率按项目分别报告、不设统一硬阈，但"半小时翻 10 行"那类文件必须从独立模块里消失。

**真实项目来源 = C**：用户提供 1-2 个真实目标迁移项目，补齐缺失的小/中/大规模档（含小文件密度、循环依赖等多样结构）。

**此门是"动翻译/扩展合并"的前置——不过不许往下做。**

## 9. 砍掉 / 推迟（避免过度设计）

**砍掉**：
- **"keep-together 架构单元"**：理由（修正版）——真正会"非循环、互不依赖也分开翻出事"的，是 **Rust 目标侧冲突**（coherence/orphan/feature），但那**不在源码图能力范围、也不是分组期能预防**，现有"整组编译真门 + compile_fixing"已在执行期兜底（见 §3）。且**机械小文件（纯类型/常量/barrel）几乎不产生 impl/trait，撞 coherence/orphan 概率最低**，是最安全的合批对象。故本设计不做 keep-together，也不扩范围去碰 Rust 侧冲突。
- 专门的 agent 规划阶段（架构单元判断）：随上条溶解。
- 多轴难度打分：无法 ground，收敛成"机械 vs 其余"二分。

**推迟到有数据**：
- 内聚加权打包（验收门说内聚差才上）。
- 跨复杂度合并（往返算下来可能更省，但有质量稀释风险，数据说更省才上）。
- 大文件自动子文件拆分。
- 失败处理：交 agent 自主决策；不发明自动重试/拆分（无依据则烧 token）；等真实失败数据再设计。

## 10. 交付拆解（批 1）

紧密相关的 CLI 工作合并为一个 PR（依据 CLAUDE.md「紧密相关小任务可合并、只要审查粒度可控」）；插件翻译路径天然独立、且必须等验收过门才做。

| | 内容 | 层 |
|---|------|----|
| **PR-1 拆解引擎** | 图保留"被用符号" + `graph interfaces --deps-of` 按符号裁剪并输出签名规模 + footprint 估算器；机械判定 predicate（替换二值 danger token）+ 危险信号→规则/定向测试映射；凸性拓扑 first-fit 合批 → composite（带循环/合批类型标记）+ 冻结计划 state 字段 + **run 守门**（合批组在 PR-2 前禁止 active dispatch、显式报错不走 SCC）；dry-run 拆解报告 + 验收门判据实现 | CLI |
| **验收活动**（非 PR） | 拿真实项目跑 PR-1、对 §8 四维度判据 | — ← **GATE** |
| **PR-2 轻量翻译路径**（过门后） | run.md 机械合批组轻量路径（一次翻完+一次编译+一次签批+content-hash 复用） | Plugin |

PR-1 建出"可冻结、可量化"的拆解；验收活动在真实项目上判它是否合理（硬前置）；PR-2 仅在验收通过后做翻译路径。

> PR-1 内部不预先按功能边界切分。仅当实现后 review 容量吃不下，才临时按"审查容量"拆成 1a（符号+footprint+分类基建）/ 1b（合批+冻结+报告）——拆点是审查容量，不是功能边界。

## 12. 决策溯源
本设计经 grilling 对抗式访谈 + 两轮 Codex 审查收敛，期间砍掉多个无依据的中间设计（失败归因分类、二分降级、union-find 分组[破坏凸性]、灰度 LLM triage、加权难度公式、keep-together 架构单元）。社区调研验证：condensation + 凸性约束 BPP-P（带依赖约束的预算装箱）为分组正解，社区发现（Louvain/Leiden）不适用；复杂度用"多指标→离散档"范式（Microsoft MI / SonarQube）但只作流程档位粗筛、不作 LLM 难度预测器；逆拓扑序翻译 + 已译 skeleton 注入（AlphaTrans/EvoC2Rust）；两端降级（大文件转人工 / 小文件合并）。Codex 审查修正：①依赖注入按符号裁剪 + footprint 为新建非复用；②机械合批组需区分于 SCC 组、走新增轻量路径；③冻结计划需新增 state 字段；④流程深度与危险信号是两件事；⑤验收门范围收窄至源码侧、明示 Rust 侧由现有编译门兜底。
