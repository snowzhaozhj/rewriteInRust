# 设计审查迭代循环（REVIEW_LOOP）

> 本文件是「审查 → 优化 → 再审查」自动循环的**唯一权威流程**。`/goal` 主会话循环与 ~5h 兜底 routine 都严格按本文件执行。要改流程只改这里。
> 本文件必须自包含——兜底 routine 会在无对话上下文的全新会话里读它执行。

---

## 0. 目标与范围

- **目标**：把 `docs/design/` 下的 Rust 迁移工作台设计，迭代到「成熟开源项目级别」的迁移质量与工程质量。
- **范围**：聚焦【迁移质量】+【工程级别】+【开源成熟度】。**不引入企业级**合规/数据治理/RBAC/SIEM/组织变更/TCO 等话题。
- **产出形式**：每条发现 = 问题 + 严重度 + 位置（`文件:章节`） + 文档依据 + 具体优化方案。

## 1. 角色与上下文纪律（关键）

- **主会话 = 瘦编排器**：只做调度、判定、`git commit`、写台账。**不把设计文档原文读进主上下文。**
- 审查 / 验证 / 改文档全部交给 **Workflow / subAgent** 在隔离上下文做，只把**结构化摘要**返回主会话。
- 跨轮状态以**文件**为准（`findings-ledger.md` + git log），不依赖对话记忆（防上下文压缩丢状态）。

## 2. 严重度定义

| 级别 | 含义 |
|------|------|
| **blocker** | 会导致迁移产出不可信/错误，或工具核心流程根本跑不通 |
| **high** | 严重影响迁移质量或工程可靠性，成熟开源项目必须解决 |
| **medium** | 明显改进空间，影响质量/可维护性但有 workaround |
| **low** | 打磨项 |

**盲点扫描（通读全 `docs/design/`，找 8 维度漏掉的）**：
- **BS1 实操盲点**：从"真实迁移一个开源项目会在哪崩"的角度找坑。
- **BS2 OSS 工程基线**：CI / release / dogfooding / 翻译质量回归 eval / 性能基准 / 错误信息质量 / 可复现性 等成熟 Rust OSS 工具应有的基线。
- **BS3 内部矛盾**：跨文件自相冲突、逻辑漏洞、未定义行为（含 §4 一致性检查项）。

## 4. 每轮流程（一个 round）

**前置**：刷新锁 `docs/review/.loop-lock`（写入当前 ISO 时间戳 + runner 标识）。

1. **审查 + 验证（Workflow）**：8 维度 + 3 盲点并行审查；每条发现经独立 verifier 复核——默认怀疑，必须 Read 引用位置确认"确实存在且未被充分处理"（过滤误读已处理项/臆造），校准严重度，评估方案可行性。
2. **应用修复（Workflow 内，全自动）**：把 confirmed/adjusted 发现按**目标文件**分组；每个文件派一个 fix-agent 用 **Edit 就地修改**（禁止整文件 Write，防卡死），返回 diff 摘要。medium/low 不改的，fix-agent 须在对应文档处追加「接受/不修 + 理由」。
3. **一致性检查（主会话 Bash，输出小）**：
   ```bash
   grep -rn "source-graph\.json" docs/design/ --include="*.md"
   grep -rn "PORTING\.md" docs/design/ --include="*.md"
   grep -rn "\.claude/rules" docs/design/ --include="*.md"
   grep -rn "4 步\|4步" docs/design/ --include="*.md"
   ```
   再核对跨文件引用 / 术语（State/Milestone/Sprint/Phase A-B）/ 版本号（README 第 3 行 + index.html）同步。命中残留 → 派 subAgent 修。
4. **落账**：`git commit`（`docs: 审查轮 k 简述`）；更新 `findings-ledger.md`（见 §6）；保留锁。
5. **主会话输出本轮 compact 摘要**：各严重度计数 + 改了哪些文件 + 是否新增 high+。

## 5. 护栏（硬性）

- **不自评**：写修复的 agent 不得参与该轮的审查/验证判定（reviewer/verifier 与 fix-agent 是不同 agent）。
- **不扩范围**：修复不得让范围/复杂度超出「成熟开源项目级别」，YAGNI 优先，能简化就简化（由维度 8 把关）。
- **净增量纪律（重要，R3 起收口）**：本循环目标是**提质，不是增量**。R1+R2 已累计注水 ~1288 行，**R3 起进入收口阶段：优先净删除**（去重、合并重复表述、删过时/冗余内容），目标每轮净行数**不增、尽量下降**。修复只用「精简 / 重构 / 补一句话约束」，原则上**不整段新增**。文档膨胀本身是维度 8 必须报告并扣分的问题。
- **排除"实现期实证"类发现（重要）**：审查者**不得**提出仅要求实现期才能产生的实证数据的发现——如 benchmark / 实测基准 / 性能数据 / 批大小实测 / 定量验证 / 可行性实测 等。这类问题归属设计文档已有的 **M0 假设验证(spike)** 范畴，应列入审查报告的「转 M0 spike」清单，**不得**作为设计缺陷、不得要求 fix-agent 新增正文。只有真正的**设计级缺陷**才算 finding：自相矛盾 / 定义缺失 / 选型不当 / 流程遗漏 / 跨文件不一致 / 逻辑漏洞。
- **顽固问题决断（R4 起，收敛关键）**：对**连续 ≥2 轮被 reopened 的同一 high 问题**，fix-agent **不得再原地小修**——必须二选一：(a) 给出**具体架构决策**（结论性方案，必要时落一条 MDR）；(b) 显式标注为**已接受的设计权衡**或**转 M0 spike 验证**，并附**理由 + 重评时机**。审查者对"已附理由的接受/决策/spike 转交"**不得再作为 open high 反复提出**（除非该决策本身有逻辑缺陷）。这是让顽固核心问题真正收敛（open high→0）的唯一出路。
- **有依据**：每条结论必须能在文档定位依据，禁止臆造（本仓库有确认过的幻觉先例）。
- **不破坏权威来源**：遵守 `CLAUDE.md` 的「文件权威来源」表，改动同一信息时只改权威文件并同步引用。

## 6. findings-ledger.md 格式

- 每条 finding 一行：`ID | 维度 | 严重度 | 文件:章节 | 标题 | 状态 | 首现轮 | 末更新轮`
  - 状态取值：`open` / `fixed` / `accepted` / `rejected` / `reopened`
- 每轮一行汇总：`Round k | new_high+ | open_blocker | open_high | open_medium | open_low | converged(yes/no)`

## 7. 收敛 / 终止判定

**达标（全部成立）**：
- (a) open blocker = 0 且 open high = 0；
- (b) 连续 2 轮无新增 confirmed high+ 发现；
- (c) 全部一致性检查通过；
- (d) 剩余 medium/low 均已修复或在文档有「接受」记录。

达标时主会话输出「**=== 收敛报告 ===**」逐条证明 (a)-(d)，并停止。

**兜底**：已完成第 6 轮仍未达标 → 停止，输出残留缺口清单，**禁止宣称成功**。

## 8. 锁协议（防 /goal 与 routine 撞车）

- 锁文件：`docs/review/.loop-lock`，内容 = 最近一次轮开始的 ISO 时间戳。
- 轮开始时写/刷新锁；/goal 连跑会持续刷新。
- 判定"新鲜"：`now - lock < 60min`。
- 任何 runner（/goal 或 routine）开跑前：若锁新鲜 → 有 runner 在跑，**退出**，不重复跑。

## 9. 兜底 routine 行为（仅"续跑"，不"开跑"）

~5h routine 每次 fire，按序判断：
1. 读 `findings-ledger.md`：若**已收敛** 或 **已完成第 6 轮** → 删除自身（routine）并退出。
2. 若 ledger 显示**完成轮数 = 0**（循环从未由 /goal 启动过）→ **退出，不自动开跑**（第 1 轮须由用户手动 `/goal` 启动并验证）。
3. 若锁新鲜（60min 内）→ 退出（有 runner 在跑）。
4. 否则（有已完成轮、未收敛、锁过期）→ 获取锁，按 §4 跑下一轮。

> 设计意图：routine 是**跨额度窗口的续跑兜底**，不是发起者。它只接力被触限中断的循环，绝不在用户未启动时自行开跑第一轮。
