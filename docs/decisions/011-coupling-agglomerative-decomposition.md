# MDR-011：耦合+目录双轴凝聚合并取代机械 footprint 装箱

> 状态：**已实现 + 真实项目验收通过**（codex 两轮审查通过 → 实现 → 10 项目实测均值 ~76% 缩减、不变量/确定性/内聚硬门全过，见 §7.1）。待 PR 4 视角审查。
> 推翻对象：`docs/decomposition-redesign.md` §5/§6/§7/§9/§12 的"机械小文件 footprint 装箱"。
> 一句话：拆解从"只合并可证明机械的小文件、按字节预算连续装箱"改为"**沿耦合重边 + 同目录双轴、在 DAG 约束下凝聚式合并任意文件**"，目标把模块数缩减 ≥80%（10→~2），简单项目更激进收敛到 ~1。

## 1. 为什么推翻原设计

原设计在真实 Python 项目上几乎零优化（DEC-GATE 实测：toolz 31→29、funcy 16→16、attrs 6→1[退化]）。三处根因：

1. **预算太小**：默认 `budget=2000` token（≈180 行自身源码），小到任何普通文件都装不下第二个文件 → 全留 Single。
2. **机械门过窄**：只有"纯类型/常量/barrel + 无危险信号"才允许合批。真实项目里逻辑文件占多数，机械文件被逻辑文件在拓扑序里隔开，几乎永不连续 → 合不动。
3. **零耦合不合并**：纯耦合视角下，同包内互不 import 的独立小文件（funcy 全是）耦合边=0 → 永不合并 → 一文件一模块。

用户裁决（多次明确）：
- "1 个文件拆一个模块一定不合理"——除非单文件本身很大，而大文件在大项目里是少数，**小文件占多数**。
- "10 行和 100 行没有本质区别"——footprint/规模不是分轴。
- "agent 一次翻译近千行小菜一碟"——否定"逻辑文件混批稀释质量"的顾虑，机械门失去存在理由。
- "肯定按目录/包结构 + 调用簇这种合理方式拆"——分轴是**语义/结构**（包边界 + 调用簇），不是规模。

## 2. 调研依据（两路 subAgent + web）

- **Understand-Anything**（理解类，非翻译）：主分组 = import 图上 **Louvain 社区检测**（不是字节装箱），容量上限按**文件数**（35/批），小批合并（min 3→25），孤立文件**按目录**归并，跨单元引用靠 **neighborMap**（1-hop 邻居 + 导出符号注入）解决——单元不必切得彼此独立。印证"耦合簇 + 目录"双轴。
- **abcoder**（字节，AST-for-LLM）：走 Code-RAG 运行时按需导航，**不做离线切块**；可借鉴的是 UniAST 数据模型（`Module→Package` 层级 + 双向依赖图 + 全局 Identity 寻址），且**沿用语言原生 package 边界、不二次聚类**——反向印证"包边界 + 依赖图"是稳妥地基。
- **社区最佳实践**（带出处）：调用图/依赖图驱动分块是公认做法（arXiv:2510.04905）；AST 感知切块优于固定大小（cAST）；社区检测做"图→分层分块"最权威落地是微软 GraphRAG 的 Leiden（arXiv:2404.16130），但**对源码调用图直接跑社区检测切翻译单元无公开先例**，属本项目创新点、需自建评测。
- **单元大小（证据驱动双预算）**：翻译质量在 >2-4K token / >500-800 行后明显塌方（arXiv:2504.02017；Chroma context-rot 多模型实证）。**两套预算解耦**：① 待翻译**自身代码**目标 ~300-800 行、硬上限 ~1000 行/≈12K token；② 100-200K 大窗口留给只读依赖签名/接口上下文，不塞待翻译代码。

## 3. 为什么不用 Louvain/Leiden 库（推翻原 §12"社区发现不适用"的措辞）

原 §12 说"社区发现不适用"，理由是凸性。**修正**：方向正确（耦合聚类），但实现不引 `leiden-rs`/`single-clustering`（均千级下载、不成熟），改用 **DAG 约束的耦合凝聚合并**——等效 Louvain 的高内聚效果，但满足翻译额外的硬约束：**单元间必须保持无环拓扑序**（依赖先翻、已译签名注入下游）。Louvain/Leiden 在无向图上最大化模块度，**不保证 quotient 图无环**。

**关键正确性事实**（本设计的算法根据）：单元各自凸 ≠ quotient 无环。反例：节点 1→2、3→4，U={1,4}、V={2,3} 两单元各自凸（在原始可达上无外部陷点），但 quotient 有 U↔V 环（1→2 给 U→V，3→4 给 V→U）。故**不能自由聚类后补救**，必须**沿真实边增量合并、每步在当前 quotient 上验环**。

## 4. 核心算法：目录优先两阶段凝聚（codex[7] 修订）

输入：SCC 缩点后的 DAG（复用 `seq.scc_groups`，已折叠循环依赖）。每个**非循环**缩点组初始为一个单元；循环组单独成 `Cycle`、不参与合并（§4.3）。

> **轴优先级（修订）**：原"耦合优先 + 同目录辅助"会产生合并顺序敏感（codex[7]：`pkg/a` 先跨目录跟 `other/b` 合并→`unit_dir` 失效→`pkg/c` 残留）。改为**目录优先两阶段**，与"包结构定主干、调用簇辅助并"一致，消除顺序敏感。

通用凝聚子程序 `merge_pass(候选生成器)`：
```
重复：
  cands = 候选生成器()                       # 一组无序对 {A,B}，已规范化 key(A) < key(B)
  对 cands 按 (耦合权重 desc, pair_key asc) 排序   # 紧耦合优先；pair_key 决定性破平
  取首个**通过约束**的对：
     约束① merge 后单元 self_size 之和 ≤ budget
     约束② 在**当前 quotient**上 is_convex({A,B})（不成环）
  命中 → 合并，重算 quotient succ + reach；否则本 pass 结束。
```
- `key(unit)` = 单元全部成员 NodeId **排序后**拼接串（决定性，与 `canonical_hash` 同源）。
- `pair_key({A,B})` = `(min(key(A),key(B)), max(...))`，强制规范化 → 杜绝 `(A,B)`/`(B,A)` 枚举序差异（codex[2]）。
- 耦合权重相等时**完全由 pair_key 破平**（两侧完整成员列表），无残留平局。

**阶段 1 · 包内归并**：候选 = `{A,B}` 中 A、B 同 `unit_dir`（含**零耦合**同目录对，权重 0 也是候选）。把每个目录/包收敛到 ~1 单元（受预算+凸性约束，超预算的大包在最弱耦合缝裂成几块）。→ 解决 funcy 类零耦合独立文件（同包即同 module）。

**阶段 2 · 跨包耦合并**：候选 = `{A,B}` 中**耦合权重 > 0**（不要求同目录）。把被父包重度调用的子包等紧耦合跨包单元并起来。→ 解决调用簇跨目录情形。

`unit_dir` = 单元成员的公共直接父目录；阶段 1 合并保持同目录 → `unit_dir` 不变；阶段 2 跨目录合并后 `unit_dir` 记为 `None`（此后不再回阶段 1，无顺序敏感）。

分类每个最终单元：
```
含循环缩点组            → Cycle（契约重路径，不参与合并）
单文件·self_size>budget → ManualOverBudget（仅"单个文件"超预算，转人工）
≥2 文件               → Batch（耦合/目录簇；语义已非"机械批"）
其余单文件             → Single
```

### 4.1 两条合并轴（均按耦合权重排序，仅候选域不同）
- **轴一·目录/包**（阶段 1，主干）：候选限同 `unit_dir`。**含零耦合对** → 同包独立文件也归一 module。
- **轴二·调用簇**（阶段 2，辅助）：候选限耦合>0。`耦合权重(A,B) = A、B 成员间 Imports+Calls+Extends+UsesType 边数之和`（双向计数，两端 file_path 映射回文件组；`Extends` 即继承/实现，无独立 Implements 边类型）。

### 4.2 约束②的正确性（复用现有设施）
- `is_convex(set, reach, n)`（现有）：不存在外部 x 使"set 可达 x"且"x 可达 set"。
- 在**当前 quotient**（单元为节点）上对待合并对 {A,B} 检 `is_convex`：若 A 经中间单元 X 可达 B（A⇝X⇝B），则 set={A,B} 可达 X 且 X 可达 set → 拒绝（正是会成环情形）。从 DAG 起步、每步保此性质 → 归纳保证 quotient 始终无环。
- **§3 反例的正确处理（codex[1] 修订）**：图仅 `1→2、3→4` 时，合并 `{1,4}` 本身**不**被拒（1、4 间无路径，可合，如同目录）；真正被拦的是**随后**合并 `{2,3}`——此时已存在 `{1,4}→2`（来自 1→2）与 `3→{1,4}`（来自 3→4），`is_convex({2,3})` 检到外部单元 {1,4} 同时被 {2,3} 可达且可达 {2,3} → 拒绝。最终得无环划分 `{3}→{1,4}→{2}`。即"自由聚类后补救"会错，"沿真实边、每步当前 quotient 验环"才对。
- **复杂度与规模阈值（codex[3] 两轮修订）**：每次合并后重算 `succ + transitive_reachability` 为 O(U·(U+E))，总计最坏 O(U²·(U+E))（U=单元数）。DEC-GATE 11 个真实项目 U 均 <300，实测可接受。**降级策略只能是性能兜底，绝不牺牲正确性**——早先草案"按目录分块、块间不重算全图 reach"会漏检跨块环、破坏 DAG 不变量（codex 两轮均否决），**作废**。**最终策略**：① 算法对任意 U 都**全量重算 reach、始终全图验凸**，正确性与 DAG 不变量对所有规模成立；② `U > 800` 仅**发 warning**（"单元数 N 较大，拆解可能较慢"），**不改变行为**；③ 真正的提速 = 增量 reach 更新（合并 (a,b) 时只对"经 a/b 的可达对"做并集更新，正确性等价、不验跨块环），列为 TODO，仅当 800 阈值在真实项目频繁触发才实现。即：**慢 ≠ 错；宁可慢不可错**。

### 4.3 循环组不参与合并
缩点循环组（`is_cycle`）维持独立 `Cycle` 单元、走现有契约路径（不变）。理由：循环组已是多文件单元（已有缩减），且其重型契约路径与轻量簇路径不同，合并会混淆状态机接入。列为后续可选优化。

## 5. 预算与单位

- **budget 门控对象 = 单元成员的"自身源码 token 之和"**（`self_size`，token≈bytes/4），**不含**依赖签名。依据 §2 双预算：依赖签名是只读上下文、进大窗口，不占翻译预算；且簇内依赖多为内部成员，签名本就不算外部上下文。
- **默认 budget = 12000 token**（≈1000 行自身源码，honor"近千行小菜一碟"，贴近研究硬上限而不越界）。可 `--budget` 覆盖。
- 单元仍报告 `footprint`（自身+依赖签名，供体积观测），与 budget 门控解耦。
- **"超预算"语义钉死 = self_size（codex[4]）**：`ManualOverBudget` 仅当**单个文件** self_size > budget 时由 planner 产生；`verify_invariants` 维持"数 `ManualOverBudget` 单元数"不变（planner 已按 self_size 标注，故无需给 verify 传 budget），不再有 footprint/self 双重解读歧义。CLI 告警文案"超 footprint 预算"→改为"超自身源码预算"。合批组 self_size 之和受约束①保证 ≤ budget，故合批单元永不 over-budget。

**退化情形显式声明（codex[6]）**：缩减依赖"同目录"或"有耦合边"二者其一。**全部文件互不 import 且各在不同目录**（无阶段 1 候选、无阶段 2 候选）时，算法**正确地**退回一文件一模块——此为真·独立模块，无合理合并依据。这与"简单项目趋近 1"**不矛盾**：简单项目的文件通常同包（同目录），由阶段 1 收敛。报告中 `residual_single_file` 高 + 缩减率低时需人工区分"算法没合动"vs"本就该独立"。

## 8. 待校准 / 已知限制（验收数据定，不预拍；不静默丢）
- budget 最终值（12000 起步，按缩减率/内聚回归微调）。
- 阶段 2 是否需"同目录加权 bonus"细化（首版：阶段 1 全包候选 + 阶段 2 纯耦合，已无 bonus 概念）。
- **大扁平目录性能（PR 4 视角主审 #1，已加 U>800 warning 兜底）**：阶段 1 在含 K 文件的单目录里每轮重建 C(K,2) 候选 + 重排 + 重算可达，约 O(K³ log K)。K~上千的扁平包会从秒级劣化到分钟级。增量候选维护 + 增量 reach 更新（合并 (a,b) 时只更新经 a/b 的可达对、增量增删受影响候选）为提速 TODO，仅当真实项目频繁触发 U>800 才实现；正确性对所有规模已成立。
- **内聚门 Imports-only 盲区（主审 #4）**：合并轴计 Imports+Calls+Extends+UsesType，但 `weighted_mq`/`coupling_edges` 只数 Imports。纯靠 Calls 合出、批内零 Imports 的簇 `coupling_edges=0` → 内聚门真空通过、不校验该合并质量。修法 = 让 `weighted_mq` 计同口径边类型；因会改全项目内聚基线、需重验收，留作独立 PR（本 PR 维持 Imports-only，验收已在此口径下全过）。
- **decompose 语言探测与建图语言可能不一致（主审 #3）**：未配 `source_language` 时 decompose 按 `--root` 现场探测语言供 `classify_file` 报告，若与 `graph build` 时不同 → by_file_kind/danger 报告失真（不影响 plan/不变量）。属 MDR-010 D-2 引入的既有行为，留待与 graph build 共享语言元数据时统一。
