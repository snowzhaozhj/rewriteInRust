1. 项目定位

核心洞察：代码生成正在商品化。验证才是价值所在。（Code Metal CEO）

我们做什么

一套 Claude Code 插件生态（Skills + Workflows + Hooks + SubAgents），将散落在各处的迁移最佳实践（Bun PORTING.md、SafeTrans 迭代修复、DepTrans 依赖图引导、UA 知识图谱）编码成可重复执行的工作流。

我们不做什么

  • 端到端自动迁移工具（Code Metal 有 $2 亿融资做这个）
  • 替代 Claude Code（我们是增强层）
  • 通用代码翻译平台

核心价值主张

#价值说明
1方法论编码将 Bun/Claw-Code 等成功模式打包为开箱即用的工作流
2验证基础设施行为录制 → 差异测试 → 属性测试 → 模糊测试管线
3项目自适应自动画像源项目，路由到合适的迁移策略
4持久化产出物每次执行产出 PORTING.md、PARITY.md、测试集、决策记录

Phase 0: 项目画像

自动分析源码仓库生成项目画像：语言/框架检测、代码规模、模块结构、依赖图、测试现状、关键难点（并发模型、FFI、动态特性、插件系统）、依赖可替代性分析、迁移动机确认。

Phase 1: 迁移规划

自动生成 PORTING.md（20 类规则）、PARITY.md（进度跟踪）、AGENTS.md（AI 行为约束）。人工审查检查点。

Phase 2: 测试基础设施

核心原则：先迁移测试，再迁移代码。

评估原项目测试质量 → 翻译已有测试 / 录制行为 / AI 生成测试 → 在原语言中验证 → 建立黄金测试集

测试分层：L1 黄金文件, L2 属性测试 (proptest), L3 快照测试 (insta), L4 模糊测试 (cargo-fuzz), L5 变异测试 (cargo-mutants)

Phase 3: 增量迁移

顺序：依赖图拓扑排序，叶子模块优先。

单模块循环：语义分析 → 生成 3-5 候选 → 编译过滤 → 测试过滤 → 选最优 → clippy + 安全检查 → 对抗性审查 → 通过: 更新 PARITY.md / 失败: 编译器反馈 → LLM 重试（最多 3 轮）→ 仍失败: 降级为 FFI 桥接或人工介入

Phase 4: 集成验证

差异测试、性能基准（criterion）、unsafe 审计（cargo-geiger + Miri）、覆盖率验证、复杂度对比（检测"翻译膨胀"）

Phase 5: 渐进部署（可选）

Strangler Fig 模式、FFI 桥接、kill switch、双运行架构

5. 文档体系

三份核心文档

PORTING.md — 迁移规则宪法

20 类规则，覆盖类型映射、错误处理、内存管理、并发、命名约定、禁止模式、unsafe 策略、惯用法映射等。参考：Bun 的 576 行、约 300 条规则的 PORTING.md。

PARITY.md — 迁移进度跟踪

每个模块的状态跟踪：pending / translating / testing / verified / done。关联 commit hash 和测试通过率。参考：Claw-Code 的 9-lane checkpoint 系统。

AGENTS.md — AI 行为约束

AI agent 的显式规则：禁止删除文件、Git 安全规则、禁止引入未经批准的依赖、不确定时必须留 TODO、每个 unsafe 块必须有 SAFETY 注释。参考：pi_agent_rust。

7. 动机驱动的策略路由

动机迁移顺序验收标准允许"不等价"？

8. 常见陷阱

技术陷阱

陷阱说明缓解措施
逐行直译得到 Arc<Mutex<>> 满天飞的代码意图驱动：先解构语义，再用 Rust 惯用法重新实现
HashMap 迭代顺序Rust HashMap 使用随机化哈希差异测试 + 需要顺序时用 BTreeMap/IndexMap
UTF-8 vs UTF-16JS string.length 返回 UTF-16 code units专门的字符串语义测试用例
整数溢出Debug panic, Release wrappingPORTING.md 中明确溢出策略
全局状态模块级变量 → OnceLock/lazy_staticPhase 0 做全局状态审计
错误处理范式异常冒泡 ≠ Result 传播重新设计错误策略（anyhow vs thiserror）

流程陷阱

陷阱数据缓解措施
认知债务AI 代码问题率 1.7 倍，METR：开发者感觉快 20% 实际慢 19%代码认领：每个模块有人能不借助 AI 解释
审查瓶颈AI：1 万行/小时，人类审阅：200-500 行/小时控制并行度，不追求生成速度
最后 20%uutils 在 90.58% 兼容性停滞接受部分模块用 FFI 桥接
性能倒退Arc 开销可能超过 GC；通道间接层 +30%迁移后必须跑 benchmark

易忽略清单

  • 构建系统迁移（package.json → Cargo.toml，CI/CD 重配）

  • 回滚计划和双运行架构
  • 序列化格式字节级兼容性测试
  • 团队 Rust 学习曲线预算（3-6 个月）

11. 实施路线图

第一阶段：MVP（4-6 周）

目标：跑通 TypeScript → Rust 的单模块迁移
  • 实现 /migrate-init（项目画像）
  • 核心 PORTING.md 生成（TS→Rust 模板）
  • 单模块迁移循环（翻译 → 编译 → 测试 → 修复）
  • 基础验证管线（cargo check + clippy + 测试）
  • 黄金文件测试框架

第二阶段：质量提升（4-6 周）

  • 多候选生成 + 最优选择
  • 编译器反馈迭代修复（2-3 轮）
  • 属性测试（proptest 等价性验证）
  • 模糊测试（cargo-fuzz 差异对比）
  • PARITY.md 自动更新

第三阶段：多语言支持（4-6 周）

  • Python → Rust 支持（Mypy 类型提取 + PyO3 桥接）
  • 统一差异测试框架
  • 行为录制框架
  • 依赖图可视化

第四阶段：完善（持续）

  • C/C++ → Rust, Go → Rust 支持
  • Kani 集成（关键路径验证）
  • 社区驱动的规则库积累
  • 性能基准自动化
